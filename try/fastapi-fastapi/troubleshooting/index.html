1️⃣ General troubleshooting checklist¶

Step What to do Why it helps
✅ Verify installation bash\npip install --upgrade fastapi uvicorn\n Guarantees you’re on a supported version of FastAPI, Uvicorn (or another ASGI server), and their dependencies.
✅ Check Python version FastAPI requires Python 3.7+. Run python --version. Older interpreters lack required language features (e.g., async/await).
✅ Review your requirements.txt / pyproject.toml Ensure pinned versions are compatible (e.g., pydantic>=2). Prevents mismatched library versions that cause obscure errors.
✅ Run the app in development mode bash\nuvicorn main:app --reload\n The auto‑reload server prints detailed tracebacks and reloads on code changes.
✅ Look at the console / server logs Errors are printed to stdout/stderr. Use logging for more context. The exact traceback often points directly to the problematic line.
✅ Use a debugger or print statements python\nimport pdb; pdb.set_trace()\n Pausing execution lets you inspect variables and request data.
✅ Validate request/response schemas Use tools like httpie, curl, or Postman to send known‑good payloads. Isolates whether the problem is in the client request, the server logic, or the schema.
✅ Clear caches / restart the server Stop the process, delete __pycache__ folders, then restart. Stale compiled files can hide recent code changes.

Tip: Keep a small “sandbox” FastAPI project (e.g., the official tutorial app) that you know works. When something breaks, compare the failing project to the sandbox to spot differences.

2️⃣ Error‑by‑error guide¶

Below each error type you’ll find:

  • What it means – the typical cause.
  • How to fix it – concrete steps and code snippets.
  • Common pitfalls – things that often trip people up.

Note: Many of these errors are subclasses of FastAPIError. The specific subclass tells you where the problem originated (request, response, dependency, etc.).

🔹 HTTPException¶

Cause
Raised manually (or by FastAPI) when the request cannot be fulfilled – e.g., 404 Not Found, 401 Unauthorized, 400 Bad Request.

Fix
Ensure the route path and HTTP method match what the client calls.
 Return appropriate status codes and messages.

from fastapi import FastAPI, HTTPException

app = FastAPI()

@app.get("/users/{user_id}")
async def read_user(user_id: int):
    if user_id <= 0:
        raise HTTPException(status_code=400, detail="user_id must be positive")
    # pretend we looked up the user…
    if user_id != 42:
        raise HTTPException(status_code=404, detail="User not found")
    return {"user_id": user_id}

Pitfall
Forgetting to import HTTPException from fastapi (instead of starlette) can cause a NameError.

🔹 WebSocketException¶

Cause
Issues while establishing or communicating over a WebSocket (e.g., client disconnects, protocol errors).

Fix
Accept the connection before reading/writing.
 Wrap receive/send calls in try/except blocks to handle abrupt disconnects.

from fastapi import FastAPI, WebSocket, WebSocketException

app = FastAPI()

@app.websocket("/ws/chat")
async def chat(ws: WebSocket):
    await ws.accept()
    try:
        while True:
            data = await ws.receive_text()
            await ws.send_text(f"Echo: {data}")
    except WebSocketException as exc:
        print(f"WebSocket closed: {exc}")

Pitfall
Calling await ws.send_text() before await ws.accept() raises a RuntimeError.

🔹 FastAPIError¶

Cause
A generic base class for FastAPI‑specific errors (e.g., mis‑configured routers, duplicate routes).

Fix
Look at the full traceback – it will usually indicate the subclass (e.g., DuplicateRouteError).
 Verify that you haven’t added the same path+method twice.

# BAD – duplicate route
@app.get("/items")
async def list_items():
    ...

@app.get("/items")  # <-- raises FastAPIError at startup
async def another_list():
    ...

Pitfall
Running the app with --reload can hide duplicate‑route errors until the second file change triggers a reload.

🔹 DependencyScopeError¶

Cause
A dependency is declared with an incompatible scope (e.g., a request‑scoped dependency used as a singleton).

Fix
Use Depends correctly and, when needed, set use_cache=False for per‑request instances.
 For database sessions, create a dependency generator that yields a fresh session per request.

from fastapi import Depends, FastAPI
from sqlalchemy.orm import Session
from .database import SessionLocal

def get_db():
    db: Session = SessionLocal()
    try:
        yield db
    finally:
        db.close()

app = FastAPI()

@app.get("/items/")
async def read_items(db: Session = Depends(get_db)):
    return {"count": db.query(...).count()}

Pitfall
Returning a plain object (instead of a generator) from a dependency will make FastAPI treat it as a singleton, causing DependencyScopeError when the object expects per‑request state.

🔹 ValidationException¶

Cause
Pydantic validation failed for a model that isn’t directly tied to a request body (e.g., a custom validator you called manually).

Fix
Ensure the data you pass to a Pydantic model matches the field types and constraints.
 Use model = MyModel(**data) inside a try/except ValidationError block.

from pydantic import BaseModel, ValidationError

class Item(BaseModel):
    name: str
    price: float

def create_item(raw: dict):
    try:
        item = Item(**raw)
        return item
    except ValidationError as exc:
        raise HTTPException(status_code=422, detail=exc.errors())

Pitfall
Passing a list where a dict is expected (or vice‑versa) triggers this error.

🔹 RequestValidationError¶

Cause
FastAPI could not validate the incoming request body, query parameters, headers, or path parameters against the declared Pydantic models.

Fix
Double‑check the client payload matches the schema (field names, types, required/optional).
 Use curl or httpie to send a minimal, known‑good request.

http POST http://localhost:8000/items/ name="Apple" price:=1.23
  • If you need custom validation, add a dependency that runs before the endpoint.
from fastapi import Depends, HTTPException

def validate_price(price: float = Query(...)):
    if price < 0:
        raise HTTPException(status_code=400, detail="Price must be non‑negative")
    return price

Pitfall
FastAPI automatically converts query parameters to the annotated type; a typo like price=abc will raise this error.

🔹 WebSocketRequestValidationError¶

Cause
The data received over a WebSocket does not conform to the expected schema (e.g., JSON missing required fields).

Fix
Parse the incoming text as JSON and validate with a Pydantic model inside a try/except.
 Send a clear error message back to the client.

from fastapi import WebSocket
from pydantic import BaseModel, ValidationError
import json

class ChatMessage(BaseModel):
    user: str
    message: str

@app.websocket("/ws/chat")
async def chat(ws: WebSocket):
    await ws.accept()
    while True:
        raw = await ws.receive_text()
        try:
            data = json.loads(raw)
            msg = ChatMessage(**data)
            await ws.send_text(f"{msg.user}: {msg.message}")
        except (json.JSONDecodeError, ValidationError) as exc:
            await ws.send_text(f"Invalid payload: {exc}")

Pitfall
Forgetting to await ws.accept() before the first receive_text() will raise a different error, masking the validation issue.

🔹 ResponseValidationError¶

Cause
FastAPI tried to serialize the response object but it didn’t match the declared response model (e.g., missing required fields).

Fix
Ensure the endpoint returns an instance (or compatible dict) of the response model.
 If you return raw data, either remove the response_model argument or convert the data.

from fastapi import FastAPI
from pydantic import BaseModel

class UserOut(BaseModel):
    id: int
    username: str

app = FastAPI()

@app.get("/users/{uid}", response_model=UserOut)
async def get_user(uid: int):
    # BAD: missing 'username'
    return {"id": uid}
    # FIX:
    # return UserOut(id=uid, username="alice")

Pitfall
Returning a SQLAlchemy model directly (instead of a Pydantic schema) often triggers this error because the ORM object isn’t JSON‑serializable.

🔹 PydanticV1NotSupportedError¶

Cause
You’re using an older version of Pydantic v1 while FastAPI (≥0.100) expects Pydantic v2.

Fix
Upgrade Pydantic:

pip install --upgrade "pydantic>=2"

If you must stay on v1 (e.g., legacy code), pin FastAPI to a version that still supports it:

pip install "fastapi<0.100"

Pitfall
Sometimes uvicorn or other dependencies bring an older Pydantic version back in; use pip list | grep pydantic to verify the installed version.

🔹 OwnerError¶

Cause
FastAPI (or a third‑party extension) attempted to perform an operation that the current OS user does not have permission for—common when binding to privileged ports (<1024) or accessing protected files.

Fix
Run the server on a non‑privileged port (e.g., 8000) or use sudo only if you understand the security implications.
 Adjust file permissions (chmod, chown) for any config files, logs, or database sockets.

# Run on port 80 with proper privileges (use a reverse proxy instead!)
sudo uvicorn main:app --host 0.0.0.0 --port 80

Pitfall
Running the whole app as root is discouraged; instead, use a reverse proxy (nginx, Caddy) to expose port 80 and let Uvicorn listen on a high port.

🔹 InternalError¶

Cause
A catch‑all for unexpected server‑side failures (e.g., unhandled exceptions, database connection loss, third‑party library bugs).

Fix
1. Read the traceback – it points to the exact line.
2. Add proper exception handling around risky code (DB calls, external APIs).
3. Log the error with stack trace for later analysis.

import logging
from fastapi import FastAPI, HTTPException

logger = logging.getLogger("uvicorn.error")

app = FastAPI()

@app.get("/external")
async def call_external():
    try:
        # pretend this may raise a network error
        result = await some_async_http_call()
        return result
    except Exception as exc:
        logger.exception("External call failed")
        raise HTTPException(status_code=502, detail="Bad gateway")

Pitfall
Swallowing exceptions (except Exception: pass) hides the root cause and turns a useful traceback into a vague InternalError.

🔹 UnicornException¶

Cause
Issues specific to the Uvicorn ASGI server (e.g., mis‑configured SSL, invalid command‑line options, or a failure to bind the socket).

Fix
Verify the command you used to start Uvicorn.
 If you’re using SSL, ensure the certificate/key files exist and are readable.
* Check that the host/port isn’t already in use (lsof -i :8000).

# Typical dev command
uvicorn main:app --reload --host 127.0.0.1 --port 8000

# Example SSL command
uvicorn main:app --host 0.0.0.0 --port 443 \
    --ssl-keyfile ./key.pem --ssl-certfile ./cert.pem

Pitfall
Running uvicorn inside a virtual environment but launching it from a system‑wide shell can pick up the wrong binary, leading to version mismatches.
